# 公开研究 Wiki 方案文档

## 1. 目标

本文档用于明确三件事：

1. `llm-wiki-skill` 仓库本身能做什么，不能做什么
2. 当前用户需求的边界、优先级和适配方式
3. 一套可落地的技术方案，用于把本地由 agent 维护的论文知识库发布到公开域名，并持续支持 agent 检索与整理

---

## 2. 用户需求

### 2.1 已确认需求

- 用户有一个自己的域名，计划上线一个公开网页
- 用户不会直接在网页上写内容
- 用户会在自己的电脑上把论文笔记、摘录、思考交给 agent
- 这些内容需要被整理进入 wiki
- 网页上需要能浏览这些公开内容
- 本地 agent 需要能继续基于该知识库做检索、总结、对比、查询
- 高密度、强表达、成体系的长文，用户会单独写到 blog 上

### 2.2 需求本质

用户要的不是“在线 CMS”，而是：

- 一个**本地维护**的知识库系统
- 一个**公开展示**的静态网站
- 一个**持续可查询**的 agent 工作底座

### 2.3 成功标准

满足以下条件即视为方案成功：

- 用户可以持续把论文笔记交给 agent 整理
- 知识库内容以本地文件形式长期保存
- 公开域名可以展示结构化研究内容
- agent 可以在本地继续查询、总结、对比和扩写知识库
- blog 与 wiki 分工清晰，不相互覆盖

---

## 3. `llm-wiki-skill` 仓库能力边界

### 3.1 仓库擅长的事情

`llm-wiki-skill` 的核心定位是：**让 agent 持续维护一个本地结构化知识库**。

它擅长：

- 初始化本地知识库目录
- 将 PDF、Markdown、文本、网页、YouTube、公众号、知乎等素材整理进知识库
- 自动生成结构化内容
  - `wiki/entities/`
  - `wiki/topics/`
  - `wiki/sources/`
  - `wiki/synthesis/`
  - `wiki/queries/`
- 用本地 Markdown 作为长期存储格式
- 让 agent 基于已有知识库做查询、综述、对比、时间线和知识图谱
- 生成可离线打开的交互式知识图谱 HTML

### 3.2 仓库不直接解决的事情

`llm-wiki-skill` 不是完整的网站系统。它不直接提供：

- 在线编辑后台
- 网站用户系统
- 实时在线搜索服务
- 在线问答 API
- 自动把整个 wiki 渲染成成熟的公开网站
- 多人协作的内容审核流
- 云端数据库或在线内容管理面板

### 3.3 对当前需求的适配度

对当前需求，`llm-wiki-skill` 是**高适配但不完整闭环**：

- 对“本地写笔记 + agent 整理 + 本地查询”非常适配
- 对“公开域名展示知识图谱与结构化内容”部分适配
- 对“成熟公共网站体验”需要额外补一层静态站发布方案

结论：

> `llm-wiki-skill` 适合作为“知识生产和知识组织内核”，不适合作为最终公开网站的完整产品壳。

---

## 4. 当前需求与仓库边界的匹配关系

### 4.1 完全匹配的部分

- 本地笔记输入给 agent
- agent 整理到 wiki
- 本地知识长期积累
- 基于知识库做检索、问答、对比、综述
- 生成知识图谱

### 4.2 需要补充建设的部分

- 公开网站首页与导航
- wiki 页面浏览体验
- wiki 与 blog 的互链
- 域名部署流程
- 公开内容筛选规则

### 4.3 当前不建议立刻做的部分

以下能力不是当前第一阶段必须项：

- 网站在线编辑
- 网站侧实时 agent 问答
- 多租户或多人后台
- 在线向量数据库
- 云端增量写回

这些能力复杂度高，会显著扩大范围，不符合当前“先建立公开研究 wiki”的目标。

---

## 5. 推荐产品形态

推荐形态为：

### 5.1 本地知识库主库

本地目录作为唯一事实源，由 agent 持续维护。

用户输入来源：

- 本地 Markdown 笔记
- 论文 PDF
- 网页链接
- 手动摘录文本
- 对话中临时形成的研究结论

agent 输出目标：

- 结构化 wiki 页面
- 查询沉淀
- 主题页
- 实体页
- 来源摘要
- 综合分析
- 知识图谱

### 5.2 公开站点作为发布视图

公开域名不承担知识生产，只承担展示。

站点职责：

- 展示知识图谱
- 展示公开的 wiki 内容
- 提供主题入口
- 链接到对应 blog 长文
- 作为研究成果的公开浏览面

### 5.3 Blog 作为高密度表达层

blog 负责：

- 长文分析
- 强观点表达
- 完整论证
- 系列文章

wiki 负责：

- 结构化条目
- 概念关系
- 论文卡片
- 主题地图
- 可检索摘要

这两层应互相链接，但不要互相替代。

## 6. 推荐技术架构

### 6.1 架构总览

建议采用“三层结构”：

1. 本地知识生产层
2. 静态站发布层
3. 本地 agent 查询层

### 6.2 逻辑数据流

```text
用户笔记 / PDF / 链接 / 摘录
        ↓
     本地 agent
        ↓
   llm-wiki 工作流
        ↓
本地知识库（Markdown + graph artifacts）
        ↓
公开内容筛选 / 静态页面生成
        ↓
   部署到用户域名
```

同时，本地 agent 继续直接读取本地知识库：

```text
本地知识库
    ↓
agent query / digest / compare / graph / crystallize
```

### 6.3 建议目录职责

知识库侧：

- `raw/`: 原始资料
- `wiki/sources/`: 每篇论文、网页、来源的摘要页
- `wiki/entities/`: 概念、术语、作者、方法等实体页
- `wiki/topics/`: 聚合主题页
- `wiki/synthesis/`: 综合分析页
- `wiki/queries/`: 对外显著有价值的查询结果沉淀
- `wiki/knowledge-graph.html`: 图谱入口

发布侧建议额外准备一个公开站仓库或公开目录，职责如下：

- `index.html`: 公开首页
- `graph/`: 图谱页及其依赖
- `topics/`: 主题页渲染输出
- `sources/`: 论文摘要页渲染输出
- `blog-links/` 或页面内链接位：跳转 blog

### 6.4 部署形态

建议静态托管，不建议第一阶段引入服务端。

可选托管方案：

- GitHub Pages
- Netlify
- Vercel
- Cloudflare Pages
- 对象存储静态网站托管

优先建议：

- 如果内容和构建流程简单：GitHub Pages
- 如果后续可能加搜索、预览、构建流程：Netlify / Vercel / Cloudflare Pages

## 7. 一期技术方案

### 7.1 一期目标

先做一个能稳定工作的公开研究 wiki，而不是做一个复杂内容平台。

一期范围：

- 本地知识库维护
- 公开图谱展示
- 公开结构化页面展示
- 本地 agent 可持续查询
- blog 外链联动

### 7.2 一期不做

- 在线写作后台
- 公开站在线问答
- 权限系统
- 评论系统
- 全文搜索服务端
- 实时更新链路

### 7.3 一期建议实现方式

方案 A，最低风险：

- `llm-wiki` 继续作为本地知识主系统
- 公开站只先上线：
  - 首页
  - 图谱页
  - 若干主题页
  - 若干来源页
- 页面由本地脚本或静态站生成器从 `wiki/` 目录生成
- 每次更新后重新部署

这是最符合当前需求的 MVP 方案。

---

## 8. 二期技术方案

当一期稳定后，再考虑增强公开网站体验。

### 8.1 二期可追加能力

- 静态全文搜索
- 更好的主题聚合页
- 自动生成最近更新页
- tag / category / paper timeline 页面
- blog 与 wiki 的双向链接索引
- RSS / sitemap / SEO

### 8.2 二期仍不建议过早做的能力

- 公开站实时聊天问答
- 在线知识库写回
- 在线 agent 代写
- 数据库存储替代本地 Markdown

除非公开站流量和需求非常明确，否则这些都会显著拉高维护成本。

---

## 9. 网站与 Agent 的角色分工

### 9.1 网站职责

网站负责“读”：

- 浏览
- 发现
- 导航
- 图谱探索
- 公开分享

### 9.2 Agent 职责

agent 负责“建”和“查”：

- ingest 新素材
- 归纳已有笔记
- 查询知识库
- 做对比和综述
- 生成主题页
- 沉淀对话结果

### 9.3 关键原则

不要让网站承担知识生产核心职责。

真正的主工作区应始终保留在本地知识库，因为：

- 这是 `llm-wiki` 的强项
- 这是最可控的长期资产
- 这能避免把系统复杂度提前推到线上

## 10. 公开内容策略

虽然当前判断内容大多为公开信息，但仍建议建立内容发布规则。

### 10.1 适合公开的内容

- 论文摘要
- 方法对比
- 术语解释
- 研究脉络
- 公开资料的整合性总结
- 指向公开 blog 的导航页

### 10.2 需要谨慎公开的内容

- 大段原文复制
- 来源受版权限制的完整转载
- 个人尚未整理完成的草稿判断
- 可能引起误解的未经核实推断

### 10.3 建议规则

- wiki 页尽量写为摘要、关系、结构化概览
- 长篇完整表达放在 blog
- wiki 到 blog 做“延伸阅读”
- blog 回链到 wiki 的主题页或论文页

## 11. 推荐实现路线

### Phase 1

- 安装并使用 `llm-wiki`
- 初始化一个面向论文研究的本地知识库
- 验证论文 ingest、query、graph 流程
- 形成第一批 `sources / topics / entities`

### Phase 2

- 设计公开站信息架构
- 确定哪些 wiki 内容要发布
- 把图谱页和核心主题页上线到域名

### Phase 3

- 加强静态站体验
- 增加主题导航、最近更新、blog 链接聚合
- 视情况加入静态搜索

---

## 12. 最终结论

### 12.1 结论

对当前用户需求，推荐的系统定义是：

> 用 `llm-wiki-skill` 作为本地知识生产与 agent 查询内核，再叠加一个独立的静态公开站作为展示层。

### 12.2 关键判断

- `llm-wiki-skill` 足够胜任知识库内核
- 它不应被直接当作完整公开网站解决方案
- 你的需求与它的强项高度一致
- 只要补一个静态发布层，这个方案就能闭环

### 12.3 执行建议

先不要把范围扩大成“在线知识平台”。

应先落地：

- 本地知识库
- agent 工作流
- 公开图谱页
- 公开主题页
- blog 联动

等这一套稳定后，再考虑搜索、问答和自动化发布。
